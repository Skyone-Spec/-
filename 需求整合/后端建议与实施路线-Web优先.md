# 后端建议与实施路线（Web 优先）

## 一、结论

建议第一阶段明确做成：

- Web 平台
- 前后端分离
- 管理后台优先
- 学生端先做响应式 H5 / Web 页面

不建议第一阶段就做微信小程序。

## 二、为什么更适合先做 Web

### 1. 需求本身偏“后台管理系统”

当前最明确、最刚需的能力其实是：

- 知识库管理
- 党团流程配置与查询
- 数据导入
- 审批
- 操作留痕
- 权限控制

这些都天然更适合先做 Web 后台，而不是先围绕微信生态设计。

### 2. 小程序会引入额外复杂度

如果第一阶段做小程序，通常会额外增加：

- 微信登录接入
- 小程序发布和审核
- 前端双端适配
- 与微信消息、模板消息相关的限制
- 一些生态能力与后端接口绑定方式的变化

这会把团队精力从“业务是否跑通”转移到“平台接入是否搞定”。

### 3. 老师已经接受网页方案

会议里老师明确表示：

- 网页可以
- 电脑端适合复杂任务
- 可参考“微人大”的网页形态

这意味着做 Web 不违背甲方预期。

### 4. 手机场景也不一定必须靠小程序

老师说手机使用场景更多，真正含义是：

- 希望轻量操作能在手机完成

这并不等于：

- 必须做微信小程序

第一阶段完全可以通过响应式 Web 先满足：

- 查流程
- 查通知
- 查知识库
- 提交简单申请

## 三、建议的阶段划分

### Phase 1：把可交付最小闭环做出来

优先做 5 件事：

1. 账号与权限
2. 知识库与文件上传
3. 党团流程查询与提醒
4. 基础审批流
5. 管理员后台与操作日志

这一阶段目标不是“功能大全”，而是把最核心闭环跑通。

### Phase 2：补齐通知和数据导入

增加：

- Excel 批量导入学生数据
- 知识库文档导入
- 通知标签分发
- 邮件发送
- 更多角色权限范围控制

### Phase 3：谨慎扩展高风险功能

再考虑：

- 学业预警
- 成绩单 PDF 解析
- AI 辅助问答优化
- 小程序接入

## 四、后端架构建议

### 1. 权限模型不要只做两级角色

建议从现在开始按 RBAC 设计，至少支持：

- `SUPER_ADMIN`
- `COLLEGE_ADMIN`
- `COUNSELOR`
- `CLASS_ADVISOR`
- `LEAGUE_SECRETARY`
- `STUDENT`

同时再增加“数据范围”概念，例如：

- 全院
- 本年级
- 本班
- 本人

这样后续班主任、团支书、老师的数据可见范围才容易控制。

### 2. 审批流要保留动作历史，不要只存最终状态

至少拆成两类数据：

- 申请单主表
- 审批动作日志表

每一次动作都记录：

- 谁操作
- 什么动作
- 原状态
- 新状态
- 时间
- 备注

否则后续无法做撤回、重批、审计。

### 3. 学生信息要分“基础字段”和“敏感字段”

建议设计上就分层：

- 基础档案
- 扩展档案
- 敏感信息
- 状态变更记录

敏感字段统一走：

- 加密存储
- 脱敏返回
- 特殊权限查看

### 4. 知识库不要一开始就做成“纯大模型问答”

更稳妥的做法：

- 先做结构化知识条目
- 支持上传原始政策附件
- 每个知识条目绑定来源文件、发布时间、适用范围
- 问答接口优先返回标准答案和原文链接

AI 只做：

- 检索增强
- 摘要生成
- 关键词归并

不要让模型直接自由生成政策结论。

### 5. 导入能力要作为后端重点建设

第一阶段最值得做好的不是“爬虫”，而是“导入”。

建议至少支持：

- Excel 模板下载
- Excel 导入校验
- 错误行提示
- 导入结果回执
- 导入日志
- 导入人记录

因为老师已经明确数据主要来自人工提供，而不是系统直连。

### 6. 审计日志必须提前做

至少要记录两类日志：

- 管理员操作日志
- 业务关键动作日志

重点对象：

- 学生信息修改
- 审批动作
- 数据导入
- 知识库发布/下线
- 权限变更

## 五、接口层建议

你现在的后端骨架已经有这些模块：

- `auth`
- `knowledge`
- `party`
- `notice`
- `certificate`
- `academic`
- `admin`

建议下一步不是继续横向铺更多 mock 接口，而是先补这几类真实能力：

1. 角色与权限接口
2. 文件上传接口
3. 知识库增删改查接口
4. 数据导入接口
5. 审批动作历史接口
6. 操作日志查询接口

## 六、对现有项目的具体建议

结合当前 `/home/skyone/spec/student-service-platform-backend`，我建议你优先调整这几件事：

### 1. README 里的产品形态表述要改

当前 README 仍然把“小程序”放在默认形态里。建议改成：

- 学生端 Web / H5
- 管理端 Web 后台
- 微信登录、小程序接入作为后续扩展

### 2. 角色枚举需要扩充

当前从目录看，权限模型大概率还比较粗。应尽快把角色体系改为多级。

### 3. 增加审计日志实体

至少应新增：

- `operation_log`
- `approval_action_log`
- `data_import_task`
- `data_import_error`

### 4. 学业分析模块先降权

该模块可以保留接口占位，但第一阶段不要投入太多核心精力。

### 5. 微信登录接口保留占位即可

老师没有坚持必须微信登录，因此：

- `wechat-login` 可以保留
- 但不应成为主路径

## 七、推荐的第一阶段交付清单

如果你是后端负责人，我建议你推动团队把“可交付版本”收敛成下面这些：

### 必做

- 登录认证
- 多角色权限
- 学生基础信息管理
- 知识库管理与查询
- 党团流程查询
- 简单提醒
- 基础审批流
- 文件上传
- 操作日志

### 选做

- 邮件通知
- 简单标签通知
- 统计看板

### 暂缓

- 短信
- 小程序
- 自动抓取公众号
- 成绩单 PDF 智能解析
- 高风险学分核算
- AI 自动纠错

## 八、最终建议

对老师和团队统一口径时，可以直接表述为：

- 第一阶段先做 Web 版学院综合服务平台。
- 先满足老师最关心的知识库、党团流程、审批、数据留痕。
- 手机端通过响应式网页覆盖基础场景。
- 小程序和微信生态后续再评估，不作为当前阶段阻塞项。

这个方向更稳，也更适合你们现在的人力和课程项目节奏。
