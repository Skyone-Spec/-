## 一、与已有判断一致的结论

- 平台前期只面向信息学院，不直接对接“微人大”。
- 角色应细分为四级：学院领导、管理老师、班团骨干、普通学生。
- 助教原则上不开放审批权限。
- 智能问答与知识库、党团流程管理仍然是最高优先级。
- 学业预警仍然是高风险、低优先级模块，不能做成过度自动化的毕业判断。
- 审批记录与管理员操作日志必须保留。
- 学生只能看自己的数据，不能导出全量数据。
- 手机场景重要，但网页方案明确可行。

## 二、这两份新增文档补强的点

### 1. 产品形态口径更清晰

- 老师个人倾向小程序，原因是手机操作方便。
- 但会议中已明确接受“前后端分离网页”。
- 因此当前最稳妥的产品口径应是：
  - 目标兼顾手机端与电脑端
  - 第一阶段可以优先做 Web
  - 小程序不是唯一刚性形态

### 2. 权限分层表达更标准

新增纪要把四级角色说得更标准：

1. 学院领导
2. 管理老师（辅导员、班主任）
3. 班团骨干（团支书、班长）
4. 普通学生

这意味着后端需要继续从“角色枚举”升级到“角色 + 数据范围”模型。

### 3. 数据范围控制更具体

会议转写补充了更细粒度口径：

- 班主任重点关注本班和本年级
- 若需看其他数据，应走申请
- 超级管理员负责维护学生信息
- 谁上传谁维护

这说明后端下一步不能只停在角色判断，还要继续补“班级/年级/本人”的数据范围控制。

### 4. 学生信息留存要求更明确

新增文档再次确认：

- 毕业、休学、退学、转专业等状态都建议保留
- 不做广泛公开，但平台内应能登记和查询

这与当前后端扩充学生状态字段的方向一致。

### 5. 学业分析模块的风险边界更明确

新增会议转写明确指出：

- 补修、缓修、免修、课程替代逻辑非常复杂
- 一旦算错，会严重误导学生毕业决策

结论：

- 当前后端把学业分析定义为“辅助提醒 + 人工复核”是正确方向
- 不应承诺自动生成可信毕业结论

### 6. 通知渠道要求更务实

- 邮件、微信提醒优先
- 短信可只做模拟记录
- 公众号抓取不是必须，手工导入可接受

这意味着后端第一阶段继续做：

- 通知管理
- 导入任务
- 留好消息通道扩展位

而不是优先投入短信或复杂采集。

### 7. 知识库上传能力被再次强调

新增文档再次确认：

- 老师可提供初始知识库
- 后续必须支持持续上传
- 单文件建议支持到 30MB

这意味着后端后续还需要继续补：

- 文件上传接口
- 知识条目与附件关联
- 导入模板约束

## 三、对当前后端实现的影响

结合当前仓库 [student-service-platform-backend](/home/skyone/spec/student-service-platform-backend)，新增文档对后端的影响主要是“加强已有方向”，而不是推翻已有实现。

当前已经对齐的部分：

- 多级角色语义
- 管理员日志
- 审批流与审批历史
- 学生信息状态字段
- 学业分析风险提示
- 学生自服务接口
- 知识库管理接口
- 数据范围控制
- 证明申请“仅学生本人申请/查询/学生端操作”的校验
- 知识库附件上传
- 更明确的班级/年级权限边界

当前还应继续补的部分：

1. 导入任务和实际导入结果的真实数据库闭环绑定
2. Kingbase 模式下继续做联调验收，而不只停留在结构与桥接脚本
3. 学业分析模块继续维持保守边界，并补足更多人工复核提示

## 四、结论

这两份新增文档没有改变当前后端路线，反而验证了以下做法是正确的：

- 先把后端按 Web/前后端分离的可接入模式搭稳
- 不急着把学业预警做成高风险自动判断
- 权限、日志、审批、学生状态字段要优先做扎实
- 对老师高频工作的支持应优先落在知识库、流程、导入、留痕上
